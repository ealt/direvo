# Planner Agent Documentation: AGENTS.md and Skills

## Problem

EDEN planner agents (AI agents like Claude CLI running as the planner
subprocess) operate in a specialized environment with specific conventions:
symlinked databases, artifact directories, proposal protocols, git worktree
structures. Currently, planners have no structured way to learn about this
environment — the knowledge is embedded in Python helper functions
(`planner_kit.py`) and passed implicitly through the planner script.

When an AI agent runs as a planner, it needs to understand:
- What files and databases exist in its working directory
- How to read trial results and artifacts
- How to create proposals (file + database row)
- How the workspace git graph relates to trials
- The trial lifecycle and notification protocol

Without this context, the agent either needs everything spelled out in the
prompt (bloating each call), or it operates blind and relies entirely on the
Python scaffolding to mediate all interactions.

## Motivation

These files serve two audiences at two different capability levels:

1. **Tool-using planner agents** — An agent with shell/file access (e.g.,
   Claude CLI in interactive mode, or a future MCP-equipped planner) can
   use these docs and skills to directly query databases, read artifacts,
   navigate git, and create proposals without Python scaffolding.

2. **Text-only planner agents** — The current data-fitting example runs
   Claude in `-p` (print) mode, where the Python script mediates all
   environment access. Even here, the docs provide value: `AGENTS.md` is
   auto-loaded context that helps the agent understand the data it receives
   in prompts (trial metrics, artifact summaries) and write better strategies.

3. **Experiment authors** — The documentation doubles as a human-readable
   guide for setting up new planners.

The current example demonstrates the text-only pattern. The docs are designed
to also support the tool-using pattern without changes, so experiments can
graduate to more autonomous planners by giving the agent tool access rather
than rewriting documentation.

## Design Decisions

### Files live in the experiment, not the framework

**Decision**: `AGENTS.md` and `.agents/skills/` are included by the experiment
author in their planner directory, not auto-generated by bootstrap or shipped
with the eden package.

**Rationale**: Different experiments have different needs. A simple planner
might not need skills at all. A complex one might add experiment-specific
skills. The framework provides the reference implementation (in the
data-fitting example); authors copy and adapt.

**Alternative considered**: Bootstrap copies template files into `planner_root/`
at setup time. Rejected because it couples the framework to a specific
documentation format and makes it harder to customize per experiment.

### Agent-agnostic paths with agent-specific aliasing

**Decision**: Canonical files live at `AGENTS.md` and `.agents/skills/`. For
Claude CLI, the experiment author copies or symlinks to `CLAUDE.md` and
`.claude/skills/`.

**Rationale**: Different AI agents discover context in different locations.
Claude CLI auto-loads `CLAUDE.md` and discovers skills in `.claude/skills/`.
Other agents may have different conventions. The content is identical — only
the file path differs. Keeping the canonical version agent-agnostic avoids
coupling the experiment to a specific AI tool.

**Discovery contract**: These files are not auto-wired into the agent's
session. The experiment author is responsible for making them discoverable
by their chosen agent — whether that means symlinking to `.claude/`, passing
via `--append-system-prompt-file`, or configuring an agent-specific loader.
The data-fitting example does not create `.claude/` aliases; the example's
`AGENTS.md` and `.agents/skills/` are not currently wired into the Claude
session. A Claude-using experiment would symlink `CLAUDE.md → AGENTS.md`
and `.claude/skills/ → .agents/skills/` to make them auto-discovered.

### Skills as self-contained capability docs

**Decision**: Each skill is a standalone markdown file with frontmatter
(`name`, `description`) that fully documents one operation — including the
relevant schema, example commands, and tips.

**Rationale**: Skills need to be self-contained because an agent may invoke
one skill without having read the others. Each skill should give the agent
enough context to complete the operation without cross-referencing. The
frontmatter makes skills compatible with Claude CLI's skill discovery.

### AGENTS.md as the overview, skills as the reference

**Decision**: `AGENTS.md` provides the high-level orientation (directory layout,
trial lifecycle, schemas at a glance) and points to skills for detailed
operations. Skills contain the full reference for each capability.

**Rationale**: This mirrors the pattern of README → detailed docs. When
discovery is wired up (e.g., via `CLAUDE.md` symlink), the agent gets a
quick orientation from `AGENTS.md` at session start, then can invoke specific
skills when it needs to perform an operation.

## Created Files

All files are under `example/data-fitting/planner/`:

### `AGENTS.md`

High-level planner agent reference:
- Directory layout table (workspace, results.db, proposals.db, proposals/,
  artifacts/)
- Trial lifecycle (propose → dispatch → implement → evaluate → notify → react)
- Both database schemas (results.db read-only, proposals.db read-write)
- Artifact conventions (plan.md, notes.md, eval_report.json per trial)
- Pointer to `.agents/skills/` for detailed operations

### `.agents/skills/read-trial-artifacts.md`

How to find and read artifacts from completed trials:
- Path pattern: `.eden/artifacts/trial-{trial_id}/`
- Common files table with source and format
- Reading commands and interpretation guidance
- Tip: comparing plan.md vs notes.md reveals implementation fidelity

### `.agents/skills/query-trial-results.md`

How to query the results database:
- Schema with fixed + dynamic metric columns
- Useful queries: all successes, best by metric, failures, counts
- Access notes: read-only, `PRAGMA table_info` for metric discovery

### `.agents/skills/query-proposals.md`

How to query the proposals database:
- Full schema
- Status lifecycle diagram (drafting → ready → dispatched → completed,
  with retry path)
- Useful queries: by status, by slug, pending, running
- Notes on orchestrator's `BEGIN IMMEDIATE` claim protocol

### `.agents/skills/write-proposal.md`

How to create a new proposal:
- Step-by-step: choose slug → write plan.md → insert DB row
- Slug naming conventions
- Priority conventions (initial vs reactive)
- Parent commit strategy (build on best vs fresh start)
- Two-step drafting protocol for safety

### `.agents/skills/navigate-workspace.md`

How to explore the workspace git graph:
- Key concepts: HEAD, trial commits, parent commit strategy
- Useful git commands: log graph, diff against baseline, show commit
- Cross-referencing commits with trial results
- Tracing a line of improvement across trials

## Relationship to Existing Files

- **`planner-prompt.md`** (existing) — experiment-specific strategy role
  prompt, passed via `--append-system-prompt-file`. Unchanged. Complements
  `AGENTS.md` which covers the framework, not the experiment.

- **`plan.py`** (existing) — planner script using `planner_kit`. Unchanged.
  The Python helpers in `planner_kit` remain the programmatic interface; the
  docs teach the agent to go beyond what the script pre-formats.

- **`workspace/AGENTS.md`** (existing) — instructions for the trial
  *implementer* agent. Different audience — that agent implements strategies,
  this one proposes them.

## Scope Boundaries

**In scope**: Teaching the planner agent about the EDEN environment and
available operations.

**Out of scope**:
- Experiment-specific strategy guidance (that's `planner-prompt.md`)
- Implementer agent instructions (that's `workspace/AGENTS.md`)
- Evaluator instructions (eval.py is a script, not an agent)
- Framework internals (orchestrator, bootstrap, worktree management)

## Future Directions

**Structured tools over prose**: If planner autonomy becomes a priority, a
stronger mechanism than documentation is exposing `PlannerContext` operations
as structured tools — e.g., an MCP server or tool definitions that agents can
call directly. The Python API (`planner_kit.py`) already implements the
operations; the skills document them as prose. A tool layer would give agents
type-safe, validated access without needing to synthesize SQL or shell commands
from documentation. The current docs are a stepping stone: they work today and
would complement a future tool layer as human-readable reference.

## Open Questions

1. **Skill granularity** — Are five skills the right number? Should
   `query-trial-results` and `query-proposals` be merged into one
   `query-databases` skill? Or is separation better for focused invocation?

2. **Schema drift** — The database schemas are documented in both `AGENTS.md`
   and the query skills. If the schema changes, both need updating. Should
   the skills reference `AGENTS.md` for the schema instead of duplicating it?

3. **Experiment-specific skills** — Should there be guidance for experiment
   authors on how to add their own skills (e.g., a data-fitting-specific
   skill for analyzing residual patterns)?
