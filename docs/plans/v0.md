# DirEvo Implementation Plan

## Summary

This document turns the product PRD in [`docs/prds/v0.md`](../prds/v0.md)
into an implementation-ready plan.

Implementation defaults for v1:

- Language: Python 3.12+
- Quality bar: production-first
- Runtime shape: library-first package with a thin CLI
- Deployment: single Docker container
- Agent integration: real planner and real execution agents in v1
- Delivery strategy: runnable vertical slices

This plan is decision-complete for a first implementation. If the code diverges
from this document, either the code or this plan should be updated to restore a
single source of truth for implementation decisions.

## Goals

The v1 implementation must:

- run a full session from config
- coordinate N concurrent slots
- manage `results.db` and `proposals.db`
- own all git mutations in the orchestrator
- run a persistent planner integration and one-shot execution integrations
- commit code plus trial docs for each completed trial
- recover cleanly from execution failures
- stop based on configured termination criteria

The v1 implementation does not need:

- a web UI
- a network service API
- distributed or multi-host execution
- non-Docker deployment modes

- `config.py`: YAML loading and validation

- `logging.py`: structured logger setup and event helpers
- `models.py`: typed runtime models and enums
- `orchestrator.py`: session lifecycle and slot state machine
- `planner.py`: persistent planner session adapter and notifications
- `execution.py`: execution/evaluation subprocess adapters
- `git_manager.py`: all git operations
- `worktree.py`: worktree setup, cleanup, and filesystem ownership operations
- `termination.py`: `max_trials`, wall clock, target condition, convergence

## Runtime Architecture

### Entry point

Expose a thin CLI:

```sh
direvo run --config /workspace/.direvo/config.yaml
direvo doctor --config /workspace/.direvo/config.yaml
```

CLI behavior:

- `run` is the only required operational command
- `doctor` is required in v1 because the deployment model depends on Docker,
  `git`, `sqlite`, user creation, and external agent binaries
- there is no daemon mode and no background service API

### `doctor` checks

`doctor` must validate:

- config parses and passes validation
- workspace exists and is a git repo
- `git` is installed
- Python runtime can open SQLite DBs
- planner integration dependency is available
- `claude` executable is available
- configured evaluation script exists and is executable from the workspace
- required writable paths under `.direvo/` can be created

### Session ownership model

The orchestrator runs as `root` inside the container and is the only component
allowed to mutate git state.

Other users:

- `planner`
- `trial-0` through `trial-{N-1}`

The implementation must create users and set permissions at container startup,
not via manual operator steps.

## Configuration Contract

## Required config keys

```yaml
parallel_trials: 4
eval_script: "./evaluate.sh"
max_trials: 200
max_wall_time: "24h"
metrics_schema:
  test_pass_rate: real
objective:
  expr: "test_pass_rate"
  direction: "maximize"
```

## Optional config keys

```yaml
convergence_window: 20
target_condition: "test_pass_rate >= 0.95"
results_db: ".direvo/results.db"
proposals_db: ".direvo/proposals.db"
proposals_dir: ".direvo/proposals"
artifacts_dir: ".direvo/artifacts"
planner_start_timeout_sec: 60
execution_timeout_sec: 1800
evaluation_timeout_sec: 1800
sqlite_busy_timeout_ms: 5000
proposal_retry_priority_delta: 0.1
```

Validation rules:

- all relative paths resolve from the workspace root
- `parallel_trials` must be a positive integer
- `metrics_schema` keys must be valid SQLite identifiers
- `objective.expr` must reference metric columns or valid SQL expressions over
  them
- `objective.direction` must be `maximize` or `minimize`
- `convergence_window` requires `objective`
- `target_condition` is optional
- timeout values must be positive integers

If validation fails, `run` and `doctor` must exit non-zero before mutating
anything.

## Database Design

Use two SQLite databases:

- `results.db`: orchestrator writes, planner reads
- `proposals.db`: planner writes, orchestrator reads and updates

Connection rules:

- use stdlib `sqlite3`
- enable WAL mode
- set `busy_timeout` from config
- use explicit transactions for all write operations
- do not use ORM libraries

### `results.db`

Create a `trials` table with this schema contract:

| Column | Type | Notes |
| --- | --- | --- |
| `trial_id` | `INTEGER PRIMARY KEY AUTOINCREMENT` | reserved before setup |
| `commit_sha` | `TEXT` | null until commit exists |
| `parent_commits` | `TEXT` | JSON array |
| `branch` | `TEXT` | branch name |
| `status` | `TEXT` | `starting`, `success`, `error`, `eval_error` |
| `artifacts_uri` | `TEXT` | artifact path |
| `description` | `TEXT` | short summary |
| `timestamp` | `TEXT` | ISO 8601 UTC |

Add one column per configured metric.

Behavior:

- reserve `trial_id` by inserting a placeholder row with `status='starting'`
- on success or eval failure, update the reserved row
- on execution failure, update the reserved row to `error`
- never insert a second row for the same trial attempt

### `proposals.db`

Create a `proposals` table with this schema contract:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | `INTEGER PRIMARY KEY AUTOINCREMENT` | proposal identifier |
| `priority` | `REAL` | higher is better |
| `slug` | `TEXT` | branch slug |
| `parent_commits` | `TEXT` | JSON array |
| `artifacts_uri` | `TEXT` | proposal docs path |
| `status` | `TEXT` | `drafting`, `ready`, `dispatched`, `completed` |
| `created_at` | `TEXT` | ISO 8601 UTC |

Claim protocol:

1. planner writes proposal docs into a temp directory
2. planner inserts row with `status='drafting'`
3. planner moves docs into final proposal directory
4. planner updates row to `status='ready'`
5. orchestrator claims via `BEGIN IMMEDIATE`
6. orchestrator updates claimed row to `dispatched`
7. on success or eval failure, orchestrator marks proposal `completed`
8. on execution failure, orchestrator returns proposal to `ready` and lowers
   priority

No filesystem lockfiles should be used for dispatch coordination.

## Git And Worktree Policy

All git operations must shell out to `git` directly. Do not use a Python git
library.

`git_manager.py` owns:

- branch creation
- checkout
- merge setup
- add/commit
- merge abort / recovery reset
- worktree creation and removal

Standard lifecycle:

1. claim proposal
2. reserve `trial_id`
3. create `trial/{trial_id}-{slug}`
4. checkout worktree
5. optionally start a merge with `--no-commit`
6. clean inherited `.direvo/trial/`
7. copy proposal docs into `.direvo/trial/`
8. chown worktree back to `trial-{slot}`
9. run execution
10. run evaluation
11. commit all changes
12. copy trial docs into artifacts
13. update DB rows
14. notify planner

Recovery sequence:

1. abort merge/rebase if present
2. `git reset --hard HEAD`
3. `git clean -fd`
4. recreate `.direvo/trial/`
5. verify clean status
6. update reserved trial row to `error`
7. requeue proposal with lower priority
8. restore worktree ownership to `trial-{slot}`

## Planner And Execution Integrations

### Planner

Planner behavior is real in v1:

- persistent session
- runs as `planner`
- writes proposal docs
- writes `proposals.db`
- reads `results.db`
- receives one-way completion notifications from the orchestrator

If planner startup fails, the session must fail fast before any slot starts.

### Execution

Execution behavior is real in v1:

- one-shot `claude -p`
- runs as `trial-{slot}`
- receives a short direction prompt
- reads full plan from `.direvo/trial/plan.md`
- produces only file modifications and trial docs

### Evaluation

Evaluation is always a subprocess, not an agent.

Rules:

- runs as `trial-{slot}`
- executes from the worktree root
- writes any long-form output into `.direvo/trial/`
- writes structured metrics to stdout JSON
- if evaluation fails, commit code and docs anyway and record `eval_error`

Timeout behavior:

- planner start timeout uses `planner_start_timeout_sec`
- execution timeout uses `execution_timeout_sec`
- evaluation timeout uses `evaluation_timeout_sec`
- timeouts are treated as failures and logged with explicit reason fields

## Logging Contract

Use structured JSON logs to stdout and `.direvo/session.log`.

Required event names:

- `session_started`
- `proposal_claimed`
- `trial_id_reserved`
- `trial_started`
- `execution_complete`
- `evaluation_complete`
- `trial_complete`
- `trial_failed`
- `planner_notified`
- `session_ended`

Each event must include:

- timestamp
- session id or run id
- slot if applicable
- trial id if applicable
- proposal id if applicable
- stable event name

## Termination Behavior

`termination.py` owns all stop decisions.

Stop when any of:

- `max_trials` reached
- `max_wall_time` exceeded
- `target_condition` becomes true
- convergence stalls over `convergence_window`
- user interruption

Convergence rule:

- evaluate `objective.expr` across successful and `eval_error` trials that have
  a value for the objective
- compare the best value in the latest window against the best value before it
- use `objective.direction` to decide improvement
- if there is no older comparison window yet, convergence cannot trigger

User interruption behavior:

- stop dispatching new work immediately
- allow in-flight slots to finish their current trial lifecycle
- record final session termination reason as `user_interrupt`

## Delivery Plan

### Slice 1: Bootstrap

Deliver:

- package skeleton
- config loading and validation
- structured logging
- schema initialization for both DBs
- `doctor` command

Acceptance:

- `doctor` validates a toy workspace
- both DBs initialize correctly
- config validation fails fast on bad input

### Slice 2: Coordination Core

Deliver:

- proposal claim logic
- `trial_id` reservation
- slot state machine
- results/proposal row update helpers

Acceptance:

- concurrent claim test passes
- `trial_id` allocation is unique and monotonic
- proposal lifecycle transitions are correct

### Slice 3: Git/Worktree Lifecycle

Deliver:

- worktree creation
- branch setup
- trial doc copy/cleanup
- recovery sequence
- commit flow

Acceptance:

- temp-repo integration tests validate DAG shape
- recovery returns worktree to a clean state
- trial docs are copied and committed correctly

### Slice 4: Real Process Integrations

Deliver:

- planner adapter
- execution adapter
- evaluation adapter
- notification flow

Acceptance:

- planner starts and receives notifications
- execution subprocess runs in slot-owned worktree
- eval failure records `eval_error` without dropping code changes

### Slice 5: Container Hardening

Deliver:

- Dockerfile
- container startup logic
- user creation and permission setup
- end-to-end local run path

Acceptance:

- `docker run ... direvo run --config ...` works against a toy repo
- permission tests confirm trial users cannot access forbidden paths

## Test Plan

Create test fixtures under `tests/` for:

- temporary git repositories
- temporary worktrees
- temporary SQLite DBs
- fake planner subprocess
- fake execution subprocess
- fake evaluation script

Required automated tests:

- config validation
- schema initialization
- atomic proposal claim under concurrency
- `trial_id` reservation
- proposal lifecycle `drafting -> ready -> dispatched -> completed`
- worktree setup and trial doc lifecycle
- merge setup and conflict recovery path
- execution crash / timeout recovery
- eval failure path
- objective-based convergence behavior
- permission boundary checks
- end-to-end containerized smoke test

## Assumptions

- Python packaging and dependency management can be chosen during scaffold, but
  the runtime stays Python-first
- test doubles are allowed in automated tests, but production code uses real
  planner and execution integrations
- no backward compatibility is required because this repository has no existing
  implementation
