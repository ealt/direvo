# Plan: Data-Fitting Demo with Real AI Agents

## Context

The direvo test suite has a deterministic seed-sum e2e test (`tests/fixtures/experiment/`) that validates the orchestration machinery but uses trivial scripts, not real AI agents. We need a realistic demo that showcases direvo on a meaningful task with actual AI CLI tools (Claude for planning, Codex for implementation). This demo serves as a template for new users and validates the full orchestration flow — proposal generation, parallel dispatch, multi-metric evaluation, and reactive planning — with real AI agents.

**Scope of validation**: Running locally (not in Docker), the demo exercises orchestration, scoring, and the grant/commit/eval lifecycle. It does NOT validate filesystem permission boundaries — user switching (`su trial-{slot}`) only activates when running as root inside the Docker container with `runtime.py`-created users. True permission isolation (test.npz hidden from the implementer via Unix permissions) requires running in a privileged Docker container. No automated privileged test for this demo exists today (the existing `scripts/run_privileged_validation.sh` only runs `tests/test_permission_boundary.py`). The demo config is structured so that permission enforcement would activate in that environment, but adding a privileged test for it is out of scope for this plan.

The demo is complementary to the existing e2e test, not a replacement. It lives at `example/data-fitting/` (not under `tests/fixtures/`).

## Task

**Data fitting**: given synthetic data with `y = 0.5*sin(2x) + 0.1*x² + noise`, produce a model that maximizes R² on held-out test data. The ground truth combines a periodic and polynomial component, so the optimal approach requires discovering both.

## Files to Create

### 1. `example/data-fitting/generate_data.py`

Uses numpy. Fixed seed for reproducibility. Generates both npz files to `SCRIPT_DIR`. Run once, commit the outputs.

- `ground_truth(x) = 0.5 * sin(2x) + 0.1 * x²`
- x ∈ [-3, 3], noise σ=0.2
- 150 train, 50 test
- npz format: arrays `x` and `y` via `np.savez`

### 2. `example/data-fitting/train.npz` + `test.npz`

Pre-generated by `generate_data.py` and committed. Config validator needs `train.npz` to exist for `file_permissions` validation. npz is the natural format since both producer (generate_data.py) and consumers (eval.py, model.py) use numpy.

### 3. `example/data-fitting/.direvo/config.yaml`

```yaml
planner_root: "./planner"
workspace: "./workspace"
parallel_trials: 3
evaluate_command: "python3 eval.py"
implement_command: "codex exec --full-auto --ephemeral 'Implement the plan in .direvo/trial/plan.md'"
plan_command: "python3 plan.py"
max_trials: 15
max_wall_time: "30m"
implement_timeout_sec: 600
evaluation_timeout_sec: 60
metrics_schema:
  r_squared: real
  rmse: real
objective:
  expr: "r_squared"
  direction: "maximize"
file_permissions:
  - path: "train.npz"
    grant: "implementer"
```

Key decisions:
- **`implement_command`**: Direct `codex exec` invocation with a minimal prompt that just points at `plan.md`. All persistent context (interface contract, constraints, data format) lives in `AGENTS.md` in the workspace, which codex reads automatically. This keeps the command short and puts durable context where it belongs — in the repo, not in a config string.
- **`evaluate_command`**: `python3 eval.py` → `_resolve_command` resolves `eval.py` to absolute path since it exists at experiment_root. CWD is worktree, but eval.py path is absolute. Same for `plan_command` resolved against planner_root.
- **`file_permissions`**: train.npz granted to implementer (symlinked into worktree). test.npz NOT granted — prevents the AI from overfitting to the held-out set.
- **Multi-metric schema**: `metrics_schema` defines two columns in results.db — `r_squared` and `rmse`. Both are recorded for every successful trial. `objective.expr` references only `r_squared` for optimization (used by the orchestrator for convergence checks and `target_condition`). `rmse` is informational — the planner can read it from results.db to give Claude richer context, and it appears in session logs, but the orchestrator doesn't optimize on it. This demonstrates that `metrics_schema` can carry auxiliary metrics beyond the objective.

### 4. `example/data-fitting/eval.py`

Runs as root (user=None), CWD=worktree. Uses `Path(__file__).resolve().parent` to find train.npz and test.npz at experiment_root. Dynamically imports `model.py` from CWD (worktree) via `importlib.util`. Uses numpy throughout (`np.load` for data, numpy for metrics).

- `r_squared(y_true, y_pred)`: numpy R² computation
- `rmse(y_true, y_pred)`: numpy RMSE computation
- `main()`: load data with `np.load`, import model, call `predict()`, output JSON `{"r_squared": <float>, "rmse": <float>}`. Both keys match the `metrics_schema` column names — the orchestrator stores each into results.db. Only `r_squared` drives the objective; `rmse` is recorded as an auxiliary metric.
- On error (missing model.py, import failure, wrong shape): exit non-zero. The orchestrator marks the trial as EVAL_ERROR. The planner queries only `status = 'success'`, so broken implementations are invisible to it — no sentinel value needed, no risk of confusing a real negative R² with a failure marker.

### 5. `example/data-fitting/planner/workspace/model.py`

Baseline model committed to the workspace git repo:

```python
import numpy as np

def predict(X_train, y_train, X_test):
    """Predict the mean of training y values."""
    return np.full_like(X_test, np.mean(y_train), dtype=float)
```

R² ≈ 0.0 (constant prediction). Maximum headroom for improvement.

### 6. `example/data-fitting/planner/workspace/AGENTS.md`

Interface contract and constraints for the AI implementer. **Intentionally does not describe the task goal** (e.g., "maximize R²" or "fit data") — only the interface, constraints, and instruction to implement plan.md. If the agent knows the optimization objective, it may ignore the assigned plan and freelance its own solution. The planner controls strategy; the implementer executes it.

Contents: function signature for `predict()`, numpy-only constraint, data format for `train.npz`, and a directive to read `.direvo/trial/plan.md`.

### 7. `example/data-fitting/planner/plan.py`

Persistent subprocess following the exact pattern from `tests/fixtures/experiment/planner/plan.py`:
- `connect_db()`, `create_proposal()`, `get_trial()` — same DB access patterns
- stdin notification loop parsing `"Trial completed. ID: {trial_id}"`
- Structured logging via `DIREVO_LOG_DIR`

**Startup**: Creates `parallel_trials + 2 = 5` initial proposals with hardcoded diverse strategies:
1. Linear regression (np.linalg.lstsq)
2. Polynomial degree 3
3. Polynomial degree 5
4. Fourier features [sin(x), cos(x), sin(2x), cos(2x), x, x², 1]
5. RBF kernel regression

**Reactive**: On trial completion:
1. Query results.db for the completed trial's `r_squared`, `rmse`, and `commit_sha`
2. Query all completed trials for history
3. Read `plan.md` artifacts from `.direvo/artifacts/trial-{id}/plan.md` (available via symlink)
4. Format history: `"Trial N (R²=X.XX, RMSE=X.XX): <strategy_summary>"` — includes both metrics so Claude can reason about fit quality from two angles
5. Call `claude -p "..."` with history to get next strategy
6. Create proposal parented on the **best trial's commit** (elitism). If no successful trials exist yet (all failed), fall back to HEAD as parent.
7. Fallback if claude unavailable: generic "improve on the best" prompt

### 8. `pyproject.toml` — add numpy to dev deps

Add `"numpy>=1.24"` to `[project.optional-dependencies] dev`. This ensures the demo's eval.py and model.py unit tests run in CI without skipping. numpy is a lightweight, ubiquitous dependency for a dev-only extra.

### 9. `tests/test_data_fitting_demo.py`

Two test levels:

**Always runs** (numpy is in dev deps, so always available in CI):
- `test_r_squared_perfect` / `test_r_squared_mean` / `test_r_squared_negative`: verify eval.py's R² math
- `test_rmse_perfect` / `test_rmse_known_error`: verify eval.py's RMSE math
- `test_baseline_model_predicts_mean`: verify model.py baseline
- `test_eval_with_baseline_model`: run eval.py end-to-end with the baseline model against committed data, verify R² ≈ 0 and RMSE is positive
- `test_data_generation_determinism`: run generate_data.py twice, verify identical output

**Gated by env var** (`DIREVO_RUN_AI_DEMO=1` + claude/codex on PATH):
- Full orchestrator integration with real AI tools
- Assert: some trials succeed, best R² > 0, planner created reactive proposals

## Critical Implementation Details

1. **Command resolution**: `_resolve_command` (config.py:147-162) replaces each shlex token with its absolute path if a file exists at `experiment_root / token`. This is why `python3 eval.py` works even though CWD is the worktree — `eval.py` gets resolved to `/abs/path/eval.py`. The `codex` token won't match any file, so it passes through.

2. **Grant lifecycle**: orchestrator.py:365-381 creates train.npz symlink in worktree before implement, removes it before commit. test.npz is never granted.

3. **Eval runs after commit**: orchestrator.py:401-411 commits implementation changes, THEN runs eval as root (user=None). Eval reads the committed model.py from CWD. Grant symlinks are already removed.

4. **Planner reads artifacts via symlink**: orchestrator.py:78 symlinks `experiment_root/.direvo/artifacts → planner_root/.direvo/artifacts`. The planner's `read_artifact_plan(trial_id)` reads `.direvo/artifacts/trial-{id}/plan.md` from its own scope.

5. **Test patching**: Like test_e2e.py:234, replace `python3` with `sys.executable` in config so eval.py/plan.py use the test venv's Python (which has numpy + direvo deps).
