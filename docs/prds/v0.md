# EDEN — Product Requirements Document

## Context

We want a system that automates iterative experimentation on a code workspace. Everything runs inside a single Docker container: an orchestrator process manages a planner (persistent Claude Code session) and N execution agents (one-shot Claude Code calls) that carry out experimental trials in parallel. Each trial modifies code and commits the result, growing a git DAG where branches represent lines of investigation.

The key insight is that research is a *search over program space*. We parallelize that search by running multiple agents simultaneously on different worktrees of the same repo, with an async model where each slot independently cycles through trials as fast as it can.

## System Invariants

These hold at all times during a session:

1. Only the orchestrator (root) mutates git state
2. Each slot operates on exactly one worktree at a time
3. Each completed trial produces exactly one commit
4. A proposal is dispatchable only when status = `ready` (docs fully written)
5. Artifacts are copies, not the source of truth (git commits are canonical)
6. The orchestrator never writes document content — only copies/moves files
7. Execution agents cannot access `.git/`, `.eden/`, or other worktrees

---

## Core Concepts

| Term | Definition |
| --- | --- |
| **Session** | One full run of the system: boot → trials → teardown |
| **Trial** | A single plan → execute → evaluate cycle producing one commit |
| **Slot** | An index 0..N-1 identifying a worktree and its associated Linux user |
| **Planner** | A persistent Claude Code session (via Agent SDK). Writes proposals and updates them as results come in |
| **Proposal** | A bundle of plan documents + a SQL row. The planner creates them; the orchestrator claims the highest-priority `ready` one when a slot is free |
| **Trial Docs** | Research-paper-like documents (hypothesis, implementation notes, results) committed to git as part of each trial and copied to artifacts |
| **Config** | A YAML file specifying session parameters: N, termination criteria, metrics schema, evaluation script path, etc. |

## State Machines

### Proposal States

```txt
drafting → ready → dispatched → completed
                  ↑            |
                  └── (retry) ←┘  (on trial failure, returned to ready
                                    with lower priority)
```

- `drafting`: Planner is writing docs. Row exists but docs may be incomplete. Not visible for dispatch.
- `ready`: Docs are fully materialized. Dispatchable by orchestrator.
- `dispatched`: Claimed by a slot. Trial in progress.
- `completed`: Trial finished (successfully or with eval_error).

On trial failure, the orchestrator moves the proposal back to `ready` with decreased priority. There is no terminal `failed` state for proposals — the planner can remove stale proposals by deleting rows when it judges a direction is no longer worth pursuing.

### Trial States (per slot)

```txt
idle → setting_up → executing → evaluating → committing → recording → idle
                  → recovering → idle  (on any failure)
```

### Slot States

```txt
idle → busy → idle
```

A slot is `busy` from the moment a proposal is claimed until results are recorded (or recovery completes).

## Requirements

### R1: Orchestrator

The orchestrator is the main process inside the Docker container. **It runs as root**, which gives it unrestricted access to all worktrees, `.git/`, and `.eden/` regardless of file ownership. This is necessary because the orchestrator must read/write worktrees owned by trial users (for git operations, doc copying, and cleanup).

Responsibilities:

- Read session config, initialize `results.db` schema (planner initializes `proposals.db`)
- Create Linux users (`planner`, `trial-0` … `trial-{N-1}`) and set file permissions
- Create and manage N persistent git worktrees
- **Async dispatch**: when a slot finishes, claim the next `ready` proposal (via atomic SQLite transaction) and start a new trial
- Before each trial: create branch/merge, checkout in worktree, clean trial docs, copy plan docs from proposal into worktree
- After each trial: run evaluation, commit everything (code + docs), copy docs to artifacts directory, record results in DB
- Notify the planner when trials complete (sends trial ID via Agent SDK)
- Log all lifecycle events
- Decide when to stop (based on config)

### R2: Git as the Workspace

The code workspace is a git repo. The user provides the initial state (seed commit, project structure, test suite, etc.) — that setup is out of scope.

**The history is a DAG, not just a tree.** Trials can:

- Branch from *any* commit in history (not just leaves — backtracking is valid)
- Merge two branches (combining innovations from separate lines of work)

**Branching model:**

```txt
main (seed)
 ├── trial/001-add-data-loader
 │    ├── trial/003-improve-loader-perf
 │    └── trial/004-add-caching
 ├── trial/002-refactor-model
 │    └── trial/005-add-attention ──┐
 │                                  ├── trial/008-merge-attention-caching
 ├── trial/004-add-caching ─────────┘
 └── trial/006-try-transformer  (dead end, agent backtracked to main)
```

Branch naming: `trial/{trial_id}-{slug}` (`trial_id` from the `trials` table).

### R3: Trial Lifecycle

Each trial proceeds through these steps. The orchestrator drives all transitions; agents only operate within their designated phase.

#### Step 1 — Claim Proposal

When a slot becomes free, the orchestrator claims the highest-priority `ready` proposal using an atomic SQLite transaction on `proposals.db`:

```sql
BEGIN IMMEDIATE;
SELECT id, slug, parent_commits, artifacts_uri
  FROM proposals WHERE status = 'ready'
  ORDER BY priority DESC LIMIT 1;
UPDATE proposals SET status = 'dispatched' WHERE id = ?;
COMMIT;
```

`BEGIN IMMEDIATE` acquires a write lock on `proposals.db`, preventing two slots from claiming the same proposal. Both the orchestrator and planner write to `proposals.db`, so the lock also serializes against concurrent planner inserts. If no `ready` proposals exist, the orchestrator waits (polling or signaled by the planner).

#### Step 2 — Reserve Trial ID

The orchestrator atomically reserves a `trial_id` by inserting a placeholder row into the `trials` table:

```sql
INSERT INTO trials (status, timestamp) VALUES ('starting', datetime('now'));
-- trial_id = last_insert_rowid()
```

This ID is used for the branch name, artifact directory, log correlation, and the final results row. SQLite's `AUTOINCREMENT` guarantees uniqueness even under concurrent slot dispatch.

#### Step 3 — Set Up Worktree

The orchestrator (as root):

1. Creates branch `trial/{trial_id}-{slug}` from the parent commit
2. Checks out the branch in the slot's worktree
3. If merging, runs `git merge --no-commit` (leaves conflicts for the agent)
4. **Cleans inherited trial docs**: `rm -rf worktrees/wt-{slot}/.eden/trial/*`
5. **Copies proposal docs**: copies the proposal's doc subdirectory contents into `.eden/trial/` in the worktree
6. **Fixes ownership**: `chown -R trial-{slot} worktrees/wt-{slot}/`

#### Step 4 — Execution

```sh
su trial-{slot} -c 'cd /workspace/worktrees/wt-{slot} && claude -p "{brief direction}"'
```

The execution agent:

- Reads `.eden/trial/plan.md` for the full plan
- Modifies code in the worktree
- If there are merge conflicts, resolves them
- May write documents to `.eden/trial/` (e.g., `implementation.md`)
- **Has no git access at all** — no reads, no writes
- **Produces no structured output** — its work is the file modifications

The `-p` prompt includes a brief direction (1-2 sentences) and tells the agent to read the plan file for details. The agent's system prompt (via CLAUDE.md) defines its role and constraints.

#### Step 5 — Evaluation

```sh
su trial-{slot} -c 'cd /workspace/worktrees/wt-{slot} && ./evaluate.sh'
```

The evaluation script may write documents to `.eden/trial/` (e.g., `results.md`). It outputs quantitative results to stdout as JSON. The keys must match the `metrics_schema` in config:

```json
{
  "test_pass_rate": 0.92,
  "latency_ms": 45,
  "memory_mb": 128
}
```

There is no required `score` field. All metric fields are user-defined in the config and become individual columns in the SQLite results DB.

#### Step 6 — Commit

The orchestrator (as root) commits **everything** — code changes + all trial docs (plan, implementation notes, evaluation results):

```sh
git -C worktrees/wt-{slot} add -A
git -C worktrees/wt-{slot} commit -m "trial/{trial_id}-{slug}: {direction}"
```

This single commit captures the full trial.

#### Step 7 — Record Results

The orchestrator:

1. Gets the commit SHA from the new commit
2. Copies `.eden/trial/` contents to `artifacts/trial-{trial_id}/`
3. Updates the placeholder row in `trials` with commit_sha, metrics, status, etc. (the row was created in Step 2 with `status = 'starting'`)
4. Updates the proposal's status to `completed` in `proposals.db`
5. Sends `"Trial completed. ID: {trial_id}"` to the planner via Agent SDK

#### Step 8 — Recovery (on failure at any step)

Deterministic cleanup algorithm:

1. **Abort any in-progress git state**: `git -C worktrees/wt-{slot} merge --abort 2>/dev/null || true` `git -C worktrees/wt-{slot} rebase --abort 2>/dev/null || true`
2. **Hard reset to branch HEAD or detached HEAD**: `git -C worktrees/wt-{slot} reset --hard HEAD`
3. **Remove untracked files**: `git -C worktrees/wt-{slot} clean -fd`
4. **Recreate trial docs directory**: `mkdir -p worktrees/wt-{slot}/.eden/trial`
5. **Verify clean state**: `git -C worktrees/wt-{slot} status --porcelain` must be empty
6. **Record failure**: update the placeholder `trials` row to `status = 'error'`
7. **Return proposal to queue**: update proposal status to `ready`, decrease priority
8. **Fix ownership**: `chown -R trial-{slot} worktrees/wt-{slot}/`

### R4: Trial Artifacts

Each trial produces a subdirectory in the artifacts directory:

```txt
.eden/
  artifacts/
    trial-001/
      plan.md              ← hypothesis and proposal (from planner)
      implementation.md    ← what the agent did (from execution agent)
      results.md           ← evaluation summary (from evaluation script)
    trial-002/
      ...
```

These are copies of the documents committed to git in each trial's `.eden/trial/` directory. The artifacts directory provides easy browsing and querying outside of git (no need to checkout each branch).

**Future consideration (out of scope for MVP)**: Embed artifact text into a vector database for semantic search by the planner.

### R5: Trial Results Database

Two SQLite databases:

- `.eden/results.db` — `trials` table (orchestrator writes, planner reads)
- `.eden/proposals.db` — `proposals` table (planner writes, orchestrator reads/updates)

Splitting the databases resolves a permissions conflict: the planner needs write access to proposals but only read access to trial results. With separate files, each can have appropriate ownership and permissions.

#### `trials` table

One row per reserved trial. Initially inserted with `status = 'starting'`, then updated to `success`, `error`, or `eval_error` when the trial finishes.

**Fixed columns:**

| Field | Type | Description |
| --- | --- | --- |
| `trial_id` | integer primary key autoincrement | Unique trial identifier |
| `commit_sha` | text | Commit SHA (null if failed before commit) |
| `parent_commits` | text | JSON array of parent commit SHAs |
| `branch` | text | Branch name |
| `status` | text | `starting`, `success`, `error`, `eval_error` |
| `artifacts_uri` | text | Path to artifacts subdirectory |
| `description` | text | Brief summary |
| `timestamp` | text | ISO 8601 |

**User-defined metric columns** (from config `metrics_schema`):

All metrics become individual columns. No required `score` field. The user's experiment instructions tell the planner which metrics to optimize.

```yaml
metrics_schema:
  test_pass_rate: real
  latency_ms: real
  memory_mb: real
```

#### `proposals` table

| Field | Type | Description |
| --- | --- | --- |
| `id` | integer primary key autoincrement | Unique proposal identifier |
| `priority` | real | Higher = more promising |
| `slug` | text | Branch name slug |
| `parent_commits` | text | JSON array of parent commit SHAs |
| `artifacts_uri` | text | Path to proposal docs subdirectory |
| `status` | text | `drafting`, `ready`, `dispatched`, `completed` |
| `created_at` | text | ISO 8601 |

**Atomic proposal claim protocol:**

1. Planner writes docs to a temp directory
2. Planner inserts a row with `status = 'drafting'`
3. Planner moves docs to final location (`proposals/proposal-{id}/`)
4. Planner updates row to `status = 'ready'`
5. Orchestrator claims with `BEGIN IMMEDIATE; SELECT … WHERE status = 'ready' ORDER BY priority DESC LIMIT 1; UPDATE … SET status = 'dispatched'; COMMIT;`

This ensures: a proposal is only visible for dispatch when docs are complete, and two slots cannot claim the same proposal.

The planner queries `results.db` via `sqlite3 -readonly` and reads/writes `proposals.db` directly. A `SKILL.md` in the workspace documents both schemas and example queries.

### R6: Planner

The planner is a **persistent Claude Code session** inside the container, managed via the Claude Code Agent SDK as a subprocess of the orchestrator.

**Why persistent**: The planner accumulates context over the session — remembering what it planned, what worked, and what to try next. This enables reasoning about diversity and reacting to results in real time.

The planner runs as a dedicated `planner` Linux user.

**Capabilities**:

- Git read access (`git log`, `git show`, `git diff`, etc.)
- SQLite read access to results (`sqlite3 -readonly .eden/results.db`)
- SQLite read/write access to proposals (`sqlite3 .eden/proposals.db`)
- Can read artifact files for detailed context on specific trials
- Writes proposal documents to `.eden/proposals/`
- No git write access

**Orchestrator → planner communication** (via Agent SDK):

One-way notifications only. The orchestrator never asks the planner for proposals — it queries the `proposals` table directly.

- `send("Trial completed. ID: {trial_id}")` → planner queries the results table, then may add new proposals, reprioritize existing ones, or mark stale proposals for removal

**Context window management**: Claude Code handles long contexts with automatic compression. If quality degrades, the orchestrator can restart the planner session with a summary of the current state.

### R7: Docker Container (Single Container, Everything Inside)

A single Docker container hosts the entire experiment. The user starts it with `docker run`. The orchestrator is the container's main process (PID 1).

```sh
docker run --name eden \
  -v {workspace_repo}:/workspace \
  {image} --config /workspace/.eden/config.yaml
```

**Linux users for isolation:**

Created by the Dockerfile or orchestrator at startup:

| User | Role | Description |
| --- | --- | --- |
| `root` | Orchestrator | Runs the main process. Full access to everything. |
| `planner` | Planner | Read `.git/`, read results DB, read artifacts, rw proposals/ |
| `trial-0` … `trial-{N-1}` | Execution agents | Read/write own worktree only |

**File permission model:**

| Path | Owner | Mode | Trial users | Planner |
| --- | --- | --- | --- | --- |
| `worktrees/wt-{slot}/` | `trial-{slot}` | 700 | Own worktree: rw | No access |
| `.git/` | root | 750 (group: planner) | No access | Read |
| `.eden/results.db` | root | 640 (group: planner) | No access | Read |
| `.eden/proposals.db` | planner | 660 (group: root) | No access | Read/write |
| `.eden/proposals/` | planner | 770 (group: root) | No access | Read/write |
| `.eden/artifacts/` | root | 750 (group: planner) | No access | Read |
| Eval script, shared deps | root | 755 | Read/execute | Read/execute |

The orchestrator runs as root and can read/write everything regardless of ownership. This resolves the permission mismatch: root performs git operations, file copying, and cleanup on worktrees owned by trial users.

**Containerization strategy tradeoff:**

| | Single container + users | N containers |
| --- | --- | --- |
| **File isolation** | OS-level (stronger) | Mount-level |
| **Process isolation** | Shared — runaway agent affects others | Per-container isolation |
| **Resource limits** | Not per-slot | Per-slot via Docker flags |
| **Management** | Simple — one container | N+1 containers |
| **Self-contained** | Everything in one place | Components spread across containers |

Single container chosen because components are cooperative and the experiment is self-contained. If process isolation becomes important later, the execution subprocess interface (`claude -p`) is the same either way.

### R8: Worktree Lifecycle

**Worktrees are persistent across trials.** Created at session boot, removed at session teardown. Between trials, the orchestrator switches the branch.

```sh
# Session boot (orchestrator as root)
git worktree add worktrees/wt-0 --detach HEAD
git worktree add worktrees/wt-1 --detach HEAD
...
chown -R trial-0 worktrees/wt-0/
chown -R trial-1 worktrees/wt-1/

# Before each trial (orchestrator as root)
git branch trial/{trial_id}-{slug} {parent_sha}
git -C worktrees/wt-{slot} checkout trial/{trial_id}-{slug}
rm -rf worktrees/wt-{slot}/.eden/trial/*
cp -r .eden/proposals/proposal-{id}/* \
      worktrees/wt-{slot}/.eden/trial/
chown -R trial-{slot} worktrees/wt-{slot}/

# After execution + evaluation (orchestrator as root)
git -C worktrees/wt-{slot} add -A
git -C worktrees/wt-{slot} commit -m "trial/{trial_id}-{slug}: {direction}"
cp -r worktrees/wt-{slot}/.eden/trial/ \
      .eden/artifacts/trial-{trial_id}/

# Session teardown
git worktree remove worktrees/wt-0
...
```

### R9: Agent Permission Restrictions

Defense-in-depth: OS file permissions (primary) + Claude Code deny rules (secondary).

**Execution agents** — no git access at all:

```json
{
  "permissions": {
    "deny": ["Bash(git *)"]
  }
}
```

**Planner** — git read allowed, git write denied:

```json
{
  "permissions": {
    "deny": [
      "Bash(git commit*)", "Bash(git push*)", "Bash(git reset*)",
      "Bash(git rebase*)", "Bash(git merge*)", "Bash(git checkout*)",
```

### R10: Session Config

```yaml
# .eden/config.yaml
parallel_trials: 4              # N: number of slots
eval_script: "./evaluate.sh"    # path to evaluation script

# Termination criteria (any one triggers shutdown)
max_trials: 200
max_wall_time: "24h"
# convergence requires an objective:
objective:
  expr: "test_pass_rate"        # column name or SQL expression
  direction: "maximize"         # "maximize" or "minimize"
convergence_window: 20          # stop if no improvement in last K trials
target_condition: "test_pass_rate >= 0.95"  # optional: SQL WHERE clause

# Metrics schema — becomes columns in the SQLite trials table
metrics_schema:
  test_pass_rate: real
  latency_ms: real
  memory_mb: real

# Paths (defaults shown)
results_db: ".eden/results.db"
proposals_dir: ".eden/proposals"
artifacts_dir: ".eden/artifacts"
```

The `objective` field defines what "improvement" means for `convergence_window`. It specifies a metric expression and a direction (`maximize` or `minimize`). The orchestrator compares the best objective value in the last K trials to the best value before that window. Without `objective`, `convergence_window` is disabled.

### R11: Async Dispatch Model

No synchronized rounds. The orchestrator maintains N slots and reads from the proposals table. The core loop per slot:

```sh
async on slot_free(slot):
  proposal = claim_proposal()               # Step 1: atomic SQLite transaction
  trial_id = reserve_trial_id()             # Step 2: INSERT into trials
  set_up_worktree(slot, proposal, trial_id) # Step 3: branch, checkout, clean, copy
  log("trial_started", trial_id, slot)
  result = run_execution(slot)              # Step 4: su trial-{slot} claude -p
  if result.failed:
    recover(slot, proposal, trial_id)       # Step 8: cleanup, update trial to error
    return
  eval_result = run_evaluation(slot)        # Step 5: su trial-{slot} ./evaluate.sh
  commit_trial(slot, trial_id)              # Step 6: git add -A && git commit
  if eval_result.failed:
    record_results(trial_id, status="eval_error")
  else:
    record_results(trial_id, eval_result)   # Step 7: update trials row
  copy_docs_to_artifacts(slot, trial_id)
  update_proposal(proposal, "completed")
  notify_planner(trial_id)
  log("trial_complete", trial_id, slot)
  if not should_terminate():
    slot_free(slot)
```

All N slots run concurrently. The planner runs independently as a persistent session, writing proposals to the DB. The orchestrator claims proposals without asking the planner.

### R12: Logging

Proper logging library (e.g., Python `structlog`, TypeScript `pino`) with structured JSON output to both console and `.eden/session.log`.

| Event | When | Key fields |
| --- | --- | --- |
| `session_started` | Boot complete | config summary |
| `trial_started` | Execution launched | trial_id, slot, branch, parents |
| `execution_complete` | Agent exited | trial_id, slot, exit_code |
| `evaluation_complete` | Eval exited | trial_id, slot, metrics |
| `trial_complete` | Results recorded | trial_id, slot, commit_sha |
| `trial_failed` | Error + recovery | trial_id, slot, error, proposal_id |
| `proposal_created` | Planner added | proposal_id, priority, slug |
| `session_ended` | Termination | reason, total_trials |

### R13: Error Handling

| Failure | Recovery |
| --- | --- |
| Execution agent crashes/times out | Run recovery algorithm (R3 Step 8), return proposal to `ready` with lower priority |
| Merge has conflicts | Not an error — execution agent resolves them |
| Evaluation script fails | Commit code changes, record as `eval_error` |
| Planner returns invalid proposal | Log, notify planner of error |
| Container dies | All in-progress trials lost. Restart container; orchestrator re-initializes slots |
| Planner context saturated | Restart planner session with state summary |

### R14: Termination Conditions

Configured in `config.yaml`. Session stops when any of:

- `max_trials` reached
- A trial satisfies `target_condition` (a SQL WHERE clause over metrics)
- `max_wall_time` exceeded
- No improvement in `objective` (in the specified direction) over the last `convergence_window` trials (requires `objective` to be defined)
- User interruption (graceful: finish in-progress trials, then stop)

## Resolved Design Decisions

- **Domain**: Domain-agnostic. Evaluation is a pluggable script per-session.
- **Evaluation**: Separate phase (between execution and commit). Generic script, not Claude Code. Outputs JSON with user-defined metric keys.
- **Planning**: Persistent Claude Code session via Agent SDK inside container. Writes proposals (doc bundles + SQL rows) to shared DB/directory. Orchestrator claims proposals via atomic SQL transaction.
- **Planner query tool**: SQLite DB queried via `sqlite3 -readonly`. SKILL.md teaches the schema.
- **Containerization**: Single Docker container for everything. Orchestrator runs as root. Linux users (`planner`, `trial-*`) enforce file isolation.
- **Proposals**: Separate `proposals.db` (writable by planner). Atomic claim protocol (`drafting → ready → dispatched → completed`). No terminal `failed` state — failed proposals return to `ready` with lower priority.
- **Metrics schema**: Fully user-defined in config. No required `score` field. All metrics become individual SQLite columns.
- **Objective**: User-defined expression + direction (maximize/minimize) for convergence detection. `convergence_window` requires an `objective`.
- **Trial ID**: Reserved atomically via SQLite INSERT before worktree setup. Used for branch name, artifacts dir, log correlation, and results row.
- **Plan delivery**: Brief direction in `claude -p` prompt + planner-authored plan docs in `.eden/trial/`.
- **Execution agents**: One-shot `claude -p` as `trial-*` users. No git access. No structured output.
- **Failed trials**: Proposal returned to `ready` with lower priority.
- **Trial documents**: Committed to git. Copied to artifacts. Cleaned from worktree between trials.
- **Merges**: Orchestrator creates merge. Agent resolves conflicts.
- **Orchestrator**: Deterministic code (not an agent). Runs as root. Never writes content.
- **Config**: YAML file specifying N, termination criteria, metrics schema, objective, eval script path.
- **Logging**: Proper logging library, structured JSON, console + file.
- **Trial completion notification**: Orchestrator sends only the trial ID. Planner queries DB for details.
- **Orchestrator language**: To be decided at implementation time.
- **Seed state**: Out of scope. User provides the initial workspace repo.
- **Vector DB**: Out of scope for MVP. Design artifacts for future embedding.

## Verification Plan

1. **Git manager tests**: create worktrees, switch branches, commit, merge, reset, clean trial docs — verify DAG correctness
2. **Trial doc lifecycle test**: docs committed, copied to artifacts, cleaned on next checkout
3. **Async dispatch test**: N=2 slots with varying times, verify independent cycling
4. **Atomic claim test**: two slots attempt to claim simultaneously, verify no double-dispatch
5. **Proposal lifecycle test**: `drafting → ready → dispatched → completed`, verify docs are not dispatchable while `drafting`
